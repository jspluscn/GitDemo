## 一、项目概述

### 1.1 项目定位

Repository Core Service 是一个基于 NAS 网络存储的 **Git 仓库管理与多人文件协作** 核心微服务。面向团队场景，提供仓库创建、文件在线编辑、多人协作冲突防护、变更管理与 Git 提交推送等完整能力。

### 1.2 技术栈

| 分类 | 技术 | 版本 |
|------|------|------|
| 语言 | Java | 21 |
| 框架 | Spring Boot | 3.2.0 |
| 微服务注册 | Eureka Client + Spring Cloud | 2023.0.0 |
| ORM | MyBatis-Plus | 3.5.5 |
| 数据库 | MySQL | 8.0 |
| 缓存/分布式锁 | Redis + Redisson | 3.24.3 |
| Git 操作 | JGit | 6.7.0 |
| 构建 | Maven | — |
| 辅助 | Lombok | — |

## 二、数据模型设计

### 2.1 数据库表结构

数据库名：`repo_db`，字符集：`utf8mb4_unicode_ci`

#### 2.1.1 用户表 `t_user`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK AUTO_INCREMENT | 用户 ID |
| username | VARCHAR(50) UNIQUE NOT NULL | 用户名（唯一） |
| password_hash | VARCHAR(255) NOT NULL | 密码哈希值 |
| email | VARCHAR(100) | 邮箱 |
| full_name | VARCHAR(50) | 真实姓名 |
| status | TINYINT DEFAULT 1 | 状态：1-正常，0-禁用 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间（自动） |

索引：`idx_username(username)`, `idx_email(email)`

#### 2.1.2 代码仓库表 `t_repository`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK AUTO_INCREMENT | 仓库 ID |
| name | VARCHAR(100) NOT NULL | 仓库名称 |
| description | VARCHAR(500) | 仓库描述 |
| owner_id | BIGINT NOT NULL | 所有者用户 ID |
| nas_path | VARCHAR(255) NOT NULL | NAS 上的绝对存储路径 |
| git_url | VARCHAR(255) | Git 远程地址（可选） |
| version | INT DEFAULT 0 | 乐观锁版本号 |
| is_deleted | TINYINT DEFAULT 0 | 逻辑删除标记 |
| created_at / updated_at | DATETIME | 时间戳 |

索引：`idx_owner(owner_id)`, `idx_name(name)`

#### 2.1.3 文件索引表 `t_file_index`

> 业务层元数据表，存储文件/文件夹的索引信息，不存储文件内容。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK AUTO_INCREMENT | 索引 ID |
| repo_id | BIGINT NOT NULL | 所属仓库 ID |
| file_path | VARCHAR(500) NOT NULL | 文件相对路径 |
| file_type | TINYINT NOT NULL | 类型：1-文件，2-文件夹 |
| file_size | BIGINT DEFAULT 0 | 文件大小（字节） |
| last_commit_id | VARCHAR(64) | 最近一次 Git Commit Hash |
| lock_token | VARCHAR(64) | 当前持有者的锁 Token |
| locked_by | BIGINT | 锁定者用户 ID |
| locked_at | DATETIME | 锁定过期时间 |
| version | INT DEFAULT 0 | 乐观锁版本号（每次修改+1） |
| is_deleted | TINYINT DEFAULT 0 | 逻辑删除标记 |
| created_at / updated_at | DATETIME | 时间戳 |

约束：`UNIQUE KEY uk_repo_path(repo_id, file_path)`  
索引：`idx_repo(repo_id)`, `idx_lock(locked_by)`

#### 2.1.4 文件编辑锁记录表 `t_file_lock`

> 辅助 Redisson 分布式锁，将编辑锁状态持久化到数据库。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK AUTO_INCREMENT | 记录 ID |
| repo_id | BIGINT NOT NULL | 所属仓库 ID |
| file_path | VARCHAR(500) NOT NULL | 文件路径 |
| user_id | BIGINT NOT NULL | 锁定用户 ID |
| lock_token | VARCHAR(64) UNIQUE NOT NULL | UUID 锁令牌 |
| expire_time | DATETIME NOT NULL | 锁过期时间 |
| created_at | DATETIME | 创建时间 |

索引：`idx_path(repo_id, file_path)`, `idx_expire(expire_time)`

#### 2.1.5 Git 提交记录缓存表 `t_commit_record`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK AUTO_INCREMENT | 记录 ID |
| repo_id | BIGINT NOT NULL | 所属仓库 ID |
| commit_hash | VARCHAR(64) NOT NULL | Git Commit Hash |
| author_id | BIGINT | 提交者用户 ID |
| author_name | VARCHAR(50) | 作者姓名 |
| message | TEXT | 提交信息 |
| commit_time | DATETIME | 提交时间 |
| parent_hash | VARCHAR(64) | 父提交 Hash |
| created_at | DATETIME | 记录创建时间 |

索引：`idx_repo_commit(repo_id, commit_hash)`, `idx_author(author_id)`

#### 2.1.6 文件变更记录表 `t_file_change`

> 用户操作（创建/更新/删除）产生的未提交变更，提交后标记为已提交。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK AUTO_INCREMENT | 变更 ID |
| repo_id | BIGINT NOT NULL | 所属仓库 ID |
| file_path | VARCHAR(500) NOT NULL | 文件路径 |
| change_type | VARCHAR(20) NOT NULL | 变更类型：ADD / MODIFY / DELETE |
| new_path | VARCHAR(500) | 新路径（仅重命名/移动时有值） |
| committed | TINYINT DEFAULT 0 | 0-未提交，1-已提交 |
| commit_id | VARCHAR(64) | 提交后的 commit hash |
| operator_id | BIGINT | 操作人用户 ID |
| created_at | DATETIME | 变更创建时间 |
| committed_at | DATETIME | 提交时间 |

索引：`idx_repo_uncommitted(repo_id, committed)`, `idx_file(repo_id, file_path)`

#### 2.1.7 NAS 存储状态监控表 `t_nas_storage_status`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK AUTO_INCREMENT | 记录 ID |
| mount_point | VARCHAR(255) UNIQUE NOT NULL | NAS 挂载点路径 |
| total_space | BIGINT | 总空间（字节） |
| free_space | BIGINT | 可用空间（字节） |
| status | TINYINT DEFAULT 1 | 1-正常，0-异常 |
| last_check_time | DATETIME | 最近检查时间 |
| updated_at | DATETIME | 更新时间 |

### 2.2 实体关系

```
User (1) ──< (N) Repository       一个用户拥有多个仓库
Repository (1) ──< (N) FileIndex   一个仓库包含多个文件/文件夹
Repository (1) ──< (N) FileChange  一个仓库有多条变更记录
Repository (1) ──< (N) CommitRecord 一个仓库有多条提交记录
User (1) ──< (N) FileLock          一个用户可持有多个文件编辑锁
FileIndex (1) ──< (N) FileLock     一个文件可有多条锁记录（历史）
```

### 3.2 模块二：文件协作（核心模块）

**基础路径**：`/api/files`

#### 3.2.1 文件操作工作流

系统采用 **"先记录变更，后统一提交"** 的工作流模式：

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  1.编辑操作   │───>│ 2.查看变更列表 │───>│ 3.勾选变更    │───>│ 4.提交&推送   │
│ 创建/更新/删除│    │ /changes     │    │ 选择性提交    │    │ /commit      │
│ 写入磁盘+记录 │    │              │    │              │    │ git add/commit│
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
```

- **阶段一（文件操作）**：创建/更新/删除仅写入 NAS 磁盘 + 更新文件索引 + 记录变更到 `t_file_change`，**不执行 git commit**
- **阶段二（查看变更）**：用户查看仓库下所有未提交的变更列表
- **阶段三（提交推送）**：用户勾选要提交的变更 → 系统执行 `git reset → git add → git commit → 可选 git push`

---

#### FR-010 创建文件或文件夹

| 项目 | 说明 |
|------|------|
| 接口 | `POST /api/files/create` |
| 入参 | `FileOperationRequest`（repoId, filePath, content, operator, fileType） |

**入参说明**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| repoId | Long | 是 | 仓库 ID |
| filePath | String | 是 | 文件/文件夹相对路径 |
| content | String | 否 | 文件内容（null 时默认空字符串） |
| operator | String | 是 | 操作人用户名 |
| fileType | Integer | 否 | 1-文件（默认），2-文件夹 |

**业务流程**：
1. 检查 NAS 健康状态 → 不健康则拒绝
2. 校验用户是否存在
3. 检查文件/文件夹是否已存在（基于 `t_file_index`） → 已存在则拒绝
4. **获取 L2 文件级分布式锁**（等待 3 秒，租约 30 秒） → 获取失败则拒绝
5. **获取 L1 仓库级分布式锁**（等待 5 秒，租约 60 秒） → 获取失败则拒绝
6. 执行操作：
   - 文件：调用 `gitService.writeFileContent()` 写入磁盘
   - 文件夹：调用 `gitService.createFolder()` 创建目录 + `.gitkeep`
7. 创建 `t_file_index` 索引记录
8. 插入 `t_file_change` 变更记录（changeType = `ADD`，committed = 0）
9. 释放 L1 仓库锁 → 释放 L2 文件锁
10. 返回 `FileChange` 对象

**返回示例**：
```json
{
  "code": 200,
  "data": {
    "id": 1,
    "repoId": 1,
    "filePath": "docs/readme.txt",
    "changeType": "ADD",
    "committed": 0
  }
}
```

#### FR-011 更新文件或文件夹

| 项目 | 说明 |
|------|------|
| 接口 | `PUT /api/files/update` |
| 入参 | `FileOperationRequest`（repoId, filePath, content, newPath, operator, expectedVersion） |

**入参说明**（在创建基础上新增）：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| content | String | 条件必填 | 新文件内容（与 newPath 至少提供一个） |
| newPath | String | 条件必填 | 新路径，用于重命名/移动 |
| expectedVersion | Integer | 否 | 客户端读取时的版本号，用于乐观锁冲突检测 |

**业务流程**：
1. 检查 NAS 健康状态
2. 校验用户存在 + 文件索引存在
3. **⭐ 乐观锁冲突检测**：若 `expectedVersion` 不为 null 且与当前 `FileIndex.version` 不一致 → 抛出 `ConflictException`（HTTP 409），提示 "File conflict detected...Please reload the file and retry."
4. **⭐ 编辑锁校验**：查询 `t_file_lock` 中是否有其他用户的活跃编辑锁 → 有则拒绝，提示 "File is currently being edited by user 'xxx'"
5. 获取 L2 文件级分布式锁
6. 获取 L1 仓库级分布式锁
7. 执行更新操作：
   - **仅更新内容**：写入新内容到磁盘
   - **仅重命名/移动**：`git rm` 旧路径 + `git add` 新路径 + 物理移动文件
   - **重命名 + 更新内容**：两者同时执行
   - **文件夹重命名**：移动所有子文件 + 更新所有子项索引
8. 更新 `t_file_index`（版本号 +1，更新人和时间）
9. 插入 `t_file_change`（changeType = `MODIFY`）
10. 释放锁，返回 `FileChange`

**冲突响应示例**（HTTP 409）：
```json
{
  "code": 409,
  "message": "File conflict detected: file 'docs/readme.txt' has been modified by another user (expected version 1, current version 3). Please reload the file and retry."
}
```

#### FR-012 删除文件或文件夹

| 项目 | 说明 |
|------|------|
| 接口 | `DELETE /api/files/delete?repoId=&filePath=&operator=` |

**业务流程**：
1. 检查 NAS + 用户 + 文件索引存在
2. **⭐ 编辑锁校验**（同 FR-011）
3. 获取 L2 文件锁 + L1 仓库锁
4. 执行删除：
   - 文件：物理删除磁盘文件
   - 文件夹：递归删除文件夹及所有子文件
5. 标记 `t_file_index` 为已删除（`is_deleted = 1`，版本号 +1）
6. 文件夹时：级联标记所有子项索引为已删除
7. 插入 `t_file_change`（changeType = `DELETE`）
8. 释放锁，返回 `FileChange`

#### FR-014 提交并推送

| 项目 | 说明 |
|------|------|
| 接口 | `POST /api/files/commit` |
| 入参 | `CommitRequest`（repoId, operator, commitMessage, selectedChangeIds, pushToRemote） |

**入参说明**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| repoId | Long | 是 | 仓库 ID |
| operator | String | 是 | 操作人用户名 |
| commitMessage | String | 否 | Git 提交信息（默认 "Commit N change(s)"） |
| selectedChangeIds | List\<Long\> | 否 | 选中的变更 ID 列表，为空时提交全部 |
| pushToRemote | Boolean | 否 | 是否推送到远程，默认 false |

**业务流程**：
1. **⭐ 获取仓库级分布式锁**（等待 10 秒，租约 120 秒） → 防止并发 commit 导致 Git 暂存区状态混乱
2. 查询待提交的 `FileChange` 记录
3. 执行 Git 操作：
   - `git reset`（重置暂存区，避免混入非预期文件）
   - 按变更类型分类：ADD/MODIFY → `git add`，DELETE → `git rm`，重命名 → 旧路径 rm + 新路径 add
   - `git commit`（使用操作人信息作为 author）
4. 标记所有变更为已提交（`committed = 1`，记录 `commit_id` 和 `committed_at`）
5. 更新相关 `FileIndex` 的 `lastCommitId`
6. 若 `pushToRemote = true`，执行 `git push`
7. 释放仓库级锁
8. 返回 commit hash

### 3.3 模块三：编辑锁管理（防并发冲突）

#### FR-020 获取编辑锁

| 项目 | 说明 |
|------|------|
| 接口 | `POST /api/files/lock/acquire` |
| 入参 | `FileLockRequest`（repoId, filePath, operator/userId, expireSeconds） |
| 默认超时 | 300 秒（5 分钟） |

**业务规则**：
- 文件未被锁定 → 创建新锁，同步更新 `t_file_index.locked_by` 和 `locked_at`
- 同一用户再次请求 → 续期已有锁
- 其他用户已持有活跃锁 → 拒绝，返回 "File is already locked by user 'xxx'"

**典型使用场景**：用户在前端打开文件编辑器时调用。

#### FR-021 释放编辑锁

| 项目 | 说明 |
|------|------|
| 接口 | `POST /api/files/lock/release` |

**业务规则**：
- 锁持有者本人释放 → 删除锁记录 + 清除 `FileIndex` 锁定状态
- 非持有者尝试释放 → 拒绝
- 无活跃锁时调用 → 静默成功（幂等）

#### FR-022 查询编辑锁状态

| 项目 | 说明 |
|------|------|
| 接口 | `GET /api/files/lock/status?repoId=&filePath=` |

**返回示例**（已锁定）：
```json
{
  "code": 200,
  "data": {
    "locked": true,
    "lockedBy": "zhangsan",
    "lockedAt": "2026-06-07T10:30:00",
    "expireTime": "2026-06-07T10:35:00",
    "lockToken": "uuid-token-xxx"
  }
}
```

**返回示例**（未锁定）：
```json
{ "code": 200, "data": { "locked": false } }
```

| 项目 | 说明 |
|------|------|
| 接口 | `GET /api/files/content?repoId=&filePath=&commitHash={可选}` |
| 逻辑 | commitHash 为空时读取工作区文件；指定 commitHash 时读取特定版本 |

## 四、并发冲突防护机制（核心设计）

系统采用 **三级防护策略** 解决多人协作场景下的文件冲突问题：

### 4.1 L1：仓库级分布式锁（Redisson）

| 属性 | 值 |
|------|------|
| 锁粒度 | 整个仓库 |
| 锁类型 | Redisson 分布式互斥锁 |
| 锁键 | `repo:lock:{repoId}:repo` |
| 用途 | 防止 Git 索引文件在并发操作时损坏 |
| 应用范围 | 创建/更新/删除/提交 所有操作均需获取 |

### 4.2 L2：文件级分布式锁（Redisson）

| 属性 | 值 |
|------|------|
| 锁粒度 | 单个文件 |
| 锁键 | `repo:lock:{repoId}:file:{filePath}` |
| 用途 | 防止同一文件的并发写入导致磁盘内容交叉覆盖 |
| 应用范围 | 创建/更新/删除 操作需获取 |

### 4.3 L3：乐观锁版本校验

| 属性 | 值 |
|------|------|
| 机制 | `FileIndex.version` 字段，每次修改 +1 |
| 用途 | 防止"后写覆盖"逻辑冲突 |
| 工作流 | 客户端读取时获取 version → 提交时通过 `expectedVersion` 传回 → 服务端校验一致性 |
| 失败响应 | HTTP 409 Conflict + 详细版本信息 |

### 4.4 L4：编辑锁（数据库持久化）

| 属性 | 值 |
|------|------|
| 存储 | `t_file_lock` 表 + `t_file_index.locked_by` |
| 过期策略 | 默认 5 分钟自动过期，支持续期 |
| 用途 | "谁在编辑"可视化 + 编辑互斥 |
| 工作流 | 打开文件 → acquire → 其他用户只读 → 保存/取消 → release |
| 校验时机 | 更新和删除操作前检查是否有其他用户的活跃编辑锁 |

### 4.5 提交阶段保护

`commitAndPush` 方法独立获取仓库级锁（等待 10 秒，租约 120 秒），确保 `git reset → git add → git commit` 流程的原子性。

### 4.6 完整协作时序

```
用户A                          服务端                          用户B
  │                              │                              │
  │── 打开文件 ──────────────────>│                              │
  │<── 获取编辑锁(lockToken) ────│                              │
  │── 读取内容(version=5) ──────>│                              │
  │<── 文件内容 ─────────────────│                              │
  │                              │<── 打开文件 ─────────────────│
  │                              │── 编辑锁被A持有，返回 ──────>│
  │                              │   "locked by userA"(仅提示)  │
  │                              │<── 读取内容(version=5) ──────│
  │                              │── 文件内容 ─────────────────>│
  │── 编辑...                    │                    编辑... ──│
  │                              │                              │
  │── 提交更新 ─────────────────>│                              │
  │   (expectedVersion=5)        │                              │
  │   版本校验: 5==5 ✅          │                              │
  │   写入磁盘, version→6        │                              │
  │<── 更新成功 ─────────────────│                              │
  │── 释放编辑锁 ───────────────>│                              │
  │                              │                              │
  │                              │<── 提交更新 ─────────────────│
  │                              │   (expectedVersion=5)        │
  │                              │   版本校验: 5≠6 ❌          │
  │                              │── 409 Conflict ────────────>│
  │                              │   "请重新加载文件"            │
```

## 九、非功能性需求

| 类别 | 要求 |
|------|------|
| **并发安全** | 三级锁 + 乐观锁 + 编辑锁，确保多人协作数据一致性 |
| **存储可靠性** | NAS 健康检查定时巡检，异常时自动拒绝写入 |
| **可观测性** | NAS 状态变化日志输出 |
| **事务一致性** | 文件操作使用 `@Transactional`，失败时回滚数据库变更 |
| **幂等性** | 编辑锁释放操作幂等，无锁时调用不报错 |
| **微服务治理** | 支持 Eureka 服务注册与发现 |
| **可扩展性** | 文件锁过期时间可配置，Git 远程推送可选 |
